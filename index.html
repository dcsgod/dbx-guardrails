dbx‑guardrails is a Databricks native guardrail system. It checks every inbound message for exposed personal data, off topic drift, and unsafe language, then hands back one typed decision: allow, mask, or block. Install it as a library or serve it as a Databricks Model Serving endpoint, same policy, same decision type, every caller.

Why it exists

Guardrails, without the sprawl

Most teams end up with a regex filter in one place, a moderation API in another, and a prompt injection check bolted on somewhere else, each with its own config file and its own opinion about what counts as unsafe. dbx‑guardrails collapses that sprawl into a single policy engine that every project on Databricks can share.

Every message passes through the same four checks in core/ before policy_engine resolves them into a single PolicyDecision.
Architecture

One decision type, two callers

GuardrailClient exposes the exact same interface whether it runs inside your Python process or talks to a served endpoint over REST. The caller never needs to know which mode is active, and both paths resolve to the same PolicyDecision, an action of allow, mask, or block.

allow – message passes untouched mask – sensitive spans redacted, message continues block – message stopped before it reaches your model
The three checks

What the policy engine actually looks for

Each guard is independent, testable on its own, and unified behind one policy so a project only has to configure it once.

Selective PII masking

Masks only the spans that actually need masking instead of redacting whole messages, so context that is safe to keep stays readable downstream.

core/pii_guard

Scope enforcement

Blocks off topic or out of context questions. Conversation history is read on request, so a terse follow up like "and last year?" is not false positived into an off scope block.

core/scope_guard

Harm detection

Flags offensive or unsafe content using a benchmark ladder of classifiers, from a fast regex baseline up to a distilled model trained on real traffic.

core/harm_guard
Getting started

Install, then pick a runtime mode

Install what you need from three optional dependency groups, then choose library mode for in process Python calls or service mode for everything else.

terminal
# core + test deps
pip install -e ".[dev]"

# + torch/transformers for training/
pip install -e ".[training]"

# + pandas/sklearn for eval/
pip install -e ".[eval]"

Library mode

in process · async

Runs inside your existing Python service. Pass recent conversation turns so scope checks understand context, or omit history and each message is judged on its own.

from client.async_client import GuardrailClient

client = GuardrailClient(project_id="supply_chain_qa", mode="library")
decision = await client.check(
    "and last year?",
    history=[
        {"role": "user", "content": "What's our OSA for produce this week?"},
        {"role": "assistant", "content": "94%, down 2pts from last week."},
    ],
)
# decision.action in {"allow", "mask", "block"}

Service mode

Databricks Model Serving · REST

Serves the same checks over REST, so a caller in any language, not only Python, gets the identical PolicyDecision shape back.

client = GuardrailClient(
    project_id="supply_chain_qa",
    mode="service",
    endpoint_url="https://<workspace>/serving-endpoints/dbx-guardrails/invocations",
)
decision = await client.check("some user message")
A full pre-check and post-check integration lives in examples/supply_chain_qa_integration.py.
Adding a project

Onboarding a new project takes zero code changes

Configuration lives in a Unity Catalog table, not in the codebase, so bringing a new project under the guardrail is a data change, not a deploy.

1

Insert a project config row

Add a row to guardrails.governance.project_configs. The table schema is defined in config/ddl/configs_table.sql and mirrored as a pydantic model in config/schema.py.

2

Let the loader pick it up

config/loader.py refreshes on a TTL cache. New or changed rows are live within scope_context_turns's TTL window, 60 seconds by default. No redeploy needed.

3

Build the scope index

Before onboarding, generate the project's in-scope example index. Omit --examples-csv to seed it with generic examples for a quick smoke test.

terminal
python -m training.scope_index.build_vector_index \
  --project-id <your_project_id> \
  --examples-csv <your_examples.csv>
Evaluation

Every model promotion runs the same ladder

python -m eval.run_benchmark runs four rungs against a seeded benign follow up regression set plus a small harm and injection sample, logging one MLflow run that covers all of them.

A model can only be promoted to champion if this run does not regress the false positive rate versus the current champion, enforced by eval.run_benchmark.promote_ok.
Under the hood

Repository layout

Every folder has exactly one job. Checking logic never needs to know how it was invoked.

core/
All checking logic: injection_gate, pii_guard, scope_guard, harm_guard, policy_engine. Zero dependency on how it is invoked.
config/
ProjectConfig schema, the Unity Catalog backed loader with its TTL cache, and the table DDL.
client/
GuardrailClient, the library mode async entry point applications call directly.
serving/
The same core/ logic wrapped as an MLflow pyfunc for Databricks Model Serving.
training/
Bootstrap label and distillation scripts for the three trained classifiers, plus the scope vector index builder.
eval/
Dataset loaders, ToxicChat, the OpenAI moderation set, HarmBench, the benign follow up regression set, and the benchmark ladder runner.
monitoring/
Lakehouse Monitoring setup and the SQL dashboard definition.
tests/
Unit tests. All model and network calls are mocked.
examples/
Reference integrations, a supply chain question and answer flow plus an async fan out example.
Where it stands

Built for real data, running on synthetic data today

  • [x] Core pipeline, config layer, client and serving wrappers
  • [x] Training scaffolding, eval harness, monitoring setup, and tests
  • [~] The three trained classifiers, PII necessity, harm, and injection, currently run on placeholder heuristics
Placeholders stay until training/*/train_classifier.py is run against real or seeded synthetic labeled data and the resulting models are registered and served. Swapping a placeholder for a trained model is a config change in serving/, not a rewrite in core/, because every check function accepts its model call as an injectable parameter.
